# Project Overview

## Portfolio Reflection
The client for this project was The Gaming Room, a company that wanted to expand its game, Draw It or Lose It, into a web-based application that could run across multiple platforms. Their goal was to move beyond a single-platform design and create software that would support multiple users, teams, and game sessions at the same time. The software also needed to enforce unique names for games and teams, support scalability, and work well in a distributed environment.

## What I Did Well
One thing I did particularly well in developing this documentation was clearly comparing different operating platforms and explaining why one option would work better than another. I also think I did a good job connecting the client’s needs to technical recommendations, especially when discussing Linux as the best server platform, scalable web architecture, memory management, storage, and security.

## What Was Helpful About the Design Document Process
Working through the design document was helpful because it forced me to think through the software before jumping into development. It made me consider system requirements, constraints, platform options, and architectural decisions early in the process. This kind of planning helps reduce confusion later because the design already outlines how the system should function and what technologies make the most sense.

## What I Would Revise
If I could revise one part of my work, I would improve the Recommendations section even more by making it more detailed and polished from the start. While I covered the required topics, I think I could make the explanations stronger by adding more depth about distributed systems, fault tolerance, and security layers. This would make the final document even more professional and complete.

## Interpreting the User’s Needs
I interpreted the user’s needs by focusing on what The Gaming Room wanted the software to accomplish: support multiple platforms, allow multiple users to play at once, maintain unique game and team names, and perform reliably as a web-based application. I used those needs to guide the design choices, such as recommending a Linux server environment, a client-server model, scalable storage, and strong security practices. Considering the user’s needs is important because the software has to solve the client’s actual problem, not just be technically impressive. My Design ApproachMy approach to designing software was to first understand the problem, then break it into smaller parts like platform requirements, system architecture, storage, memory, distributed communication, and security. From there, I compared possible solutions and chose the ones that best matched the client’s goals. In the future, I would use the same strategy by starting with requirements, analyzing constraints, comparing platform options, and then building recommendations based on performance, cost, scalability, and user needs. This process helps create a design that is practical, organized, and easier to develop.

## Final Thoughts
This project helped me better understand how important software design documentation is when building applications for real clients. It showed me that strong design is not just about code, but also about planning, communication, and making informed decisions that support both the user and the development team.
